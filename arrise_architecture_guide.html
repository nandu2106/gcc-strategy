Layer 1 — Collection

Data Collection Layer

Three signal types captured from every server: Logs (what happened), Metrics (how the system performed), and Traces (how a request travelled through services).

📋

Logs

What happened?

Timestamped text records of events. Errors, warnings, access records, application events.

Sample Log Format (JSON)
{
  "timestamp": "2025-03-11T10:23:01Z",
  "level": "ERROR",
  "service": "game-server-co-01",
  "message": "DB connection timeout",
  "trace_id": "abc123"
}
📊

Metrics

How is the system performing?

Numeric measurements over time. CPU %, memory usage, requests/sec, latency, error rates.

Sample Metric Format (Prometheus)
# HELP cpu_usage CPU usage percent
# TYPE cpu_usage gauge
cpu_usage{
 host="server-mt-07",
 region="malta"
} 87.3 1710149000
🔗

Traces

How did a request flow?

Distributed traces follow a single request across multiple services. Shows where time was spent and where failures occurred.

Sample Trace Format (Jaeger)
{
  "traceID": "abc123",
  "spans": [
   {"op": "auth", "dur": 12},
   {"op": "db.query", "dur": 340}
  ]
}

Collection Tools Explained

ToolTypeWhat it DoesRuns On
Syslog Logs OS-level system log collector. Captures kernel, auth, daemon messages. Every Linux server
Fluentd Logs Aggregates logs from multiple sources, transforms formats, routes to destinations. Log aggregator nodes
Filebeat Logs Lightweight agent that ships log files to Fluentd or OpenSearch. Low resource use. Every server (agent)
Prometheus Metrics Scrapes metrics endpoints at intervals. Stores time-series data locally. Central metrics server
Node Exporter Metrics Exposes OS/hardware metrics (CPU, memory, disk, network) for Prometheus. Every server (agent)
cAdvisor Metrics Collects container-level metrics (Docker/K8s CPU, memory per container). Every K8s node
Jaeger Traces Distributed tracing backend. Receives spans and reconstructs request journeys. Central tracing service
OpenTelemetry Traces Vendor-neutral SDK/collector. Instruments code and sends to Jaeger/Tempo. App code + collector
Tempo Traces Scalable trace storage by Grafana. Integrates with Grafana dashboards. Trace storage backend
💡 Integration Point All three signal types (Logs, Metrics, Traces) are unified using OpenTelemetry Protocol (OTLP) — a common wire format. This means a single collector can forward all signals to the ingestion layer in one standardized format.
🔗 How trace_id links everything The trace_id field appears in BOTH logs and traces. When an error log fires, engineers can look up the trace_id to replay the entire request journey — seeing exactly which service and database call caused the failure.
Layer 2 — Ingestion (AWS)

Ingestion Layer

Receives the firehose of signals from 500+ servers and routes them appropriately. Three services cover different use cases: real-time streaming, batch delivery, and event routing.

⚡

Kinesis Data Streams

Real-time streaming

Millisecond-latency ingestion. Used for critical metrics and security events that need immediate ML anomaly detection.

Record format on stream
{"PartitionKey": "server-co-01",
 "Data": "base64(OTLP payload)",
 "SequenceNumber": "..."}
🪣

Kinesis Data Firehose

Batch delivery

Buffers data (60s or 5MB batches) then delivers directly to S3. No consumer management. Great for logs that don't need real-time processing.

S3 delivery path format
s3://arrise-logs/
 year=2025/month=03/
 day=11/hour=10/
 logs-1710149000.gz
📨

MSK (Kafka) / EventBridge

Event routing

MSK is managed Kafka for high-throughput event streaming. EventBridge routes specific event patterns (e.g. critical alerts) to Lambda functions or other AWS services.

EventBridge rule pattern
{"source": ["arrise.infra"],
 "detail-type": ["CriticalAlert"],
 "detail": {"severity": ["P1"]}}

When to use which?

ScenarioBest ChoiceWhy
Detect server CPU spike in <5 seconds Kinesis Streams Real-time, feeds directly into ML inference
Archive all application logs to S3 daily Firehose Managed batching, no consumer code, auto-compresses
Route a game session event to 3 consumers MSK / Kafka Multiple consumers per topic, replay capability
Trigger a Lambda when a P1 alert fires EventBridge Pattern matching, serverless fan-out routing
Layer 3 — Storage

Tiered Storage Layer

Data is automatically moved between S3 tiers based on age. Hot data is instantly accessible; cold data is preserved cheaply for compliance and ML model retraining.

🔥

S3 Standard

Hot — 0 to 7 days

Immediately accessible. Used by Athena, Spark, and real-time ML inference. Highest cost because of frequent access.

Cost
$0.023 / GB / month
Retrieval: $0.00 (free)
Latency: milliseconds
🌡️

S3 Infrequent Access

Warm — 7 to 30 days

Lower storage cost; small retrieval fee. Used for ad-hoc investigations and weekly trend analysis.

Cost
$0.0125 / GB / month
Retrieval: $0.01 / GB
Latency: milliseconds
🧊

S3 Glacier

Cold — 30 days to 1 year+

Cheapest storage. Minutes-to-hours retrieval time. Used for compliance archival and long-term ML training datasets.

Cost
$0.004 / GB / month
Retrieval: $0.03 / GB
Latency: 1–12 hours
💡 S3 Lifecycle Policy (how data moves automatically) AWS S3 Lifecycle Rules automate tiering:

Day 0–7: S3 Standard → Day 7: transitions to S3-IA → Day 30: transitions to Glacier → Day 365+: optionally deleted or kept for compliance. Zero manual work required after setup.

Layer 4 — Processing

Processing & Analytics Layer

Transforms raw stored data into structured, enriched datasets ready for ML and dashboards. Three tools cover SQL querying, big data processing, and data transformation pipelines.

🔍

AWS Athena

SQL queries on S3

Serverless SQL engine. Query Parquet/JSON files on S3 directly — no databases to manage. Pay per query (per TB scanned).

Example query
SELECT host, AVG(cpu)
FROM s3_metrics
WHERE dt = '2025-03-11'
GROUP BY host
HAVING AVG(cpu) > 80
⚡

EMR Serverless

Spark big data jobs

Runs Apache Spark jobs at scale without managing clusters. Used for heavy ML feature engineering — computing rolling windows across billions of metric rows.

Example Spark operation
df.groupBy("host")
 .window("5min")
 .agg(avg("cpu"),
     stddev("cpu"))
 .write.parquet(s3_features)
🔄

Glue ETL

Extract, Transform, Load

Managed ETL service. Crawls S3 for schema, converts formats (JSON → Parquet), deduplicates data, and loads into structured tables for Athena.

ETL Pipeline Steps
1. EXTRACT: raw JSON from S3
2. TRANSFORM: parse, clean,
enrich with region tags
3. LOAD: write Parquet to
s3://arrise-features/
🔗 How these three work together Glue ETL runs nightly to convert raw logs → clean Parquet. Athena queries those Parquet files for ad-hoc analysis and dashboards. EMR Spark handles heavier batch computations (like computing 30-day rolling baselines for anomaly detection) that would be too slow or expensive in SQL.
Layer 5 — AI/ML

SageMaker Deep Dive

The intelligence layer. SageMaker manages the full ML lifecycle: feature engineering, model training, deployment, and real-time inference — all using the processed observability data.

1
Feature Engineering
Lambda / Glue · Runs before training & inference
▼

Raw metrics/logs/traces are not directly usable by ML models. Feature engineering transforms them into numeric vectors that capture patterns.

What happens here

  • Extract from logs/metrics/traces: Pull relevant numeric fields. E.g. extract error_count, latency_p99, cpu_mean from raw data.
  • Time-series windowing (5min, 1hr, 24hr): Compute rolling averages, standard deviations, and rates of change at multiple time scales. A CPU spike matters differently over 5min vs 24hrs.
  • Anomaly score pre-calculation: Compute z-scores (how many standard deviations from the mean) as input features to the model.
Feature vector example (one row = one server snapshot)
{
 "host": "server-mt-07",
 "cpu_mean_5m": 87.3,    // avg CPU last 5 min
 "cpu_stddev_5m": 12.1,
 "cpu_mean_1h": 65.2,    // avg CPU last 1 hr
 "error_rate_5m": 0.043,
 "latency_p99_5m": 1240, // ms
 "mem_used_pct": 91.2,
 "z_score_cpu": 3.4      // 3.4 std devs above normal
}
💡 Why multiple time windows? A server that's been at 90% CPU for 24 hours is less concerning than one that jumped from 20% to 90% in the last 5 minutes. Multi-window features capture both steady-state and change-rate patterns.

🧠 Isolation Forest vs LSTM — which when? Isolation Forest is fast, works on tabular data (current snapshot), no temporal context. Good for point-in-time anomalies.

LSTM (Long Short-Term Memory) is a recurrent neural network — it learns temporal patterns. Better at detecting anomalies that only make sense over time (e.g. memory creeping up slowly over 6 hours).
3
Model Registry & Retraining
SageMaker Pipelines · Model Monitor · A/B Testing
▼
  • Model Registry: Every trained model version is stored with metadata (accuracy, training date, dataset version). Enables rollback if a new model performs worse.
  • Model Monitor: Continuously checks if the live endpoint's predictions are drifting from training distribution. Triggers retraining when data drift is detected.
  • SageMaker Pipelines: Automates the full retrain cycle — new data lands in S3 → Glue ETL → feature engineering → training job → evaluation → deploy if better.
  • Retraining schedule: Anomaly models retrain weekly. Forecasting model retrains monthly (needs more historical data for seasonality patterns).
⚠️ Why retraining matters Infrastructure changes over time (new servers added, software versions changed, traffic patterns shift). A model trained on old data will start producing false positives or missing real anomalies. Regular retraining keeps the model aligned with current reality.

Insights & Action Layer

ML outputs are surfaced to humans (dashboards, search) and to automated systems (alerting, ticket creation). The loop closes here: insights trigger real operational actions.

🔍

Full-text search and analytics over all logs. Engineers type natural-language-style queries to investigate incidents. Also powers dashboards and log explorer UI.

📊

Grafana

Dashboards & Alerting

Real-time metric dashboards. Connects to Prometheus, Athena, and OpenSearch. Displays ML anomaly scores alongside raw metrics so engineers see context immediately.

Why Parquet for ML features?

📦

Columnar Storage

Parquet stores data column-by-column, not row-by-row. Reading just "cpu_mean" from 10M rows reads only 1 column — 100x faster than reading all columns in JSON.

🗜️

Compression

Numeric columns compress 5–10x better than JSON text. Metric time-series data that would be 100GB as JSON becomes 10GB as Parquet.

🤝

SageMaker Native

SageMaker training jobs accept Parquet directly from S3. No conversion step needed — the training job reads features in milliseconds.
